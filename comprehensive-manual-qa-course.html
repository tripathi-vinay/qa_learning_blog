üîÑ 2. SDLC (Software Development Life Cycle) & STLC (Software Testing Life Cycle)

üèóÔ∏è SDLC - Like Building a House

Think of building a house:
  1. Planning: Decide what kind of house (Requirements)
  2. Design: Create blueprints (System Design)
  3. Construction: Actually build it (Development)
  4. Inspection: Check everything works (Testing)
  5. Moving in: Start living there (Deployment)
  6. Maintenance: Fix issues, renovate (Maintenance)

üß™ STLC - The Testing Journey

Example: Testing Instagram's Photo Upload Feature

  1. Test Planning: "We need to test photo uploads on mobile and web"
  2. Test Case Design: Write scenarios like "Upload JPG file", "Upload very large file"
  3. Test Environment Setup: Get test phones, test accounts
  4. Test Execution: Actually try uploading different types of photos
  5. Bug Reporting: "Large files crash the app" - report this bug
  6. Test Closure: All critical bugs fixed, feature ready!

üìù 4. Test Case Design Techniques

üéØ Equivalence Partitioning

"Group similar inputs together"

Example: Testing Age Input on Netflix

Age groups:

  • Invalid (below 0): -5, -1
  • Child (0-12): 5, 10 ‚Üí Shows kids content
  • Teen (13-17): 15, 16 ‚Üí Shows teen content
  • Adult (18+): 25, 50 ‚Üí Shows all content
  • Invalid (too high): 200, 999

Instead of testing every age, test one from each group!

"Test the edges - that's where bugs hide!"

üé§ Interview Questions for Manual QA

üéØ Basic Questions

  1. What is the difference between SDLC and STLC?
    Answer: SDLC is the entire software development process (planning to maintenance), while STLC is specifically the testing process within SDLC.
  2. How do you prioritize test cases?
    Answer: Based on business impact, user frequency, and risk. Critical features like login/payment get highest priority.
  3. Explain the bug life cycle with an example.
  4. What is exploratory testing? Give a real scenario.
  5. How do you report a critical bug? Show me the format.

üîß Intermediate Questions

  1. What is regression testing? When do you perform it?
    Answer: Regression testing ensures that existing functionality still works after new changes. Perform it after:
    • New feature additions
    • Bug fixes
    • Code refactoring
    • Environment changes
    Example: After WhatsApp adds video calling, test that regular messaging, photo sharing, and voice calls still work properly.
  2. Difference between smoke testing and sanity testing?
    Answer:
    Aspect Smoke Testing Sanity Testing
    Purpose Check if build is stable enough for testing Verify specific functionality after minor changes
    Scope Broad - covers major functions Narrow - specific area/feature
    When After new build deployment After minor bug fixes
  3. How do you ensure maximum test coverage?
    Answer:
    • Requirements traceability: Map each requirement to test cases
    • Test case review: Peer review of test cases
    • Boundary value testing: Test edge cases and limits
    • Positive & negative scenarios: Test valid and invalid inputs
    • User journey mapping: Cover end-to-end workflows
    • Risk-based testing: Focus on high-risk areas
    Coverage Formula: (Requirements covered by tests / Total requirements) √ó 100
  4. What tools have you used for manual testing and why?
    Sample Answer:
    • JIRA: Bug tracking and project management - easy integration with development team
    • TestRail: Test case management - organized test execution and reporting
    • Postman: API testing - verify backend services work correctly
    • Browser Dev Tools: Debug frontend issues and check console errors
    • Charles Proxy: Monitor network traffic and API calls
    • Snagit: Screenshot and screen recording for bug reports
  5. How do you handle testing when requirements are unclear?
    Answer:
    • Ask questions: Clarify ambiguous requirements with stakeholders
    • Document assumptions: Write down what you understand and get approval
    • Exploratory testing: Test different scenarios to understand expected behavior
    • User story analysis: Break down requirements into smaller, testable pieces
    • Prototype testing: Test early versions to understand intended functionality
    • Create test scenarios: Write possible test cases and validate with team

üéØ Scenario-Based Questions

  1. "You found a critical bug 1 day before release. What do you do?"
    Answer:
    1. Document thoroughly: Clear steps to reproduce, impact analysis
    2. Assess severity: Does it block core functionality? Affect user data?
    3. Communicate immediately: Inform development lead, project manager, and stakeholders
    4. Provide recommendations: Fix now vs. hotfix after release vs. workaround
    5. Risk assessment: What's the impact if we release with this bug?
    6. Follow up: Track fix, retest thoroughly, regression testing
    Key Point: Present facts objectively, let stakeholders make the release decision.
  2. "Developer says 'It works on my machine.' How do you respond?"
    Answer:
    • Stay professional: "Let's work together to understand the difference"
    • Environment comparison: Compare OS, browser versions, configurations
    • Detailed reproduction: Share exact steps, test data, screenshots/videos
    • Pair testing: Reproduce the issue together on developer's machine
    • Log analysis: Check application logs, network calls, console errors
    • Environment sync: Ensure test environment matches development environment
    Remember: It's not "us vs them" - we're both trying to deliver quality software.
  3. "You have 100 test cases but only 2 hours. How do you prioritize?"
    Answer - Priority Framework:
    1. Critical Path (30 mins): Core user journeys (login, purchase, payment)
    2. High Impact (45 mins): Features most users interact with
    3. Risk Areas (30 mins): Recently changed code, complex integrations
    4. Smoke Tests (15 mins): Basic functionality validation
    Decision Matrix:
    • P1: High business impact + High usage = Must test
    • P2: Medium impact + High usage = Should test
    • P3: Low impact + Low usage = Skip for now
  4. "Describe the most challenging bug you found and how you reported it."
    Sample Answer Structure:
    • Context: "I was testing a financial app's money transfer feature"
    • Challenge: "Bug occurred only with specific amount ranges during peak hours"
    • Investigation: "I analyzed patterns, tested different scenarios, collected logs"
    • Documentation: "Created detailed reproduction steps with screenshots and network logs"
    • Impact: "Bug could have caused incorrect money transfers - high severity"
    • Resolution: "Worked with dev team to identify root cause in currency conversion logic"
    Key Elements: Show analytical thinking, communication skills, and impact understanding.
  5. "How would you test an e-commerce checkout process?"
    Comprehensive Test Approach: üõí Shopping Cart Testing:
    • Add/remove items, quantity changes, price calculations
    • Save cart between sessions, guest vs. registered user
    üí≥ Payment Testing:
    • Valid/invalid credit cards, different payment methods
    • Security validation, PCI compliance
    • Payment failures, timeouts, network issues
    üì¶ Order Processing:
    • Order confirmation emails, inventory updates
    • Shipping calculations, tax calculations
    • Order tracking, order history
    üîí Security & Performance:
    • SSL certificates, data encryption
    • Load testing during sales events
    • Cross-browser compatibility
